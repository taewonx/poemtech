# Poemtech MVP 개발 계획 및 클라이언트 AI 바이오메카닉스 텔레메트리 아키텍처

1인 창업의 핵심은 **'서버 유지비(Burn rate)의 최소화'**와 **'가장 뾰족한 하나의 기능에 집중하는 것'**입니다. Poemtech은 운동 방법을 참견하는 '코치'가 아닙니다. F1 레이서들에게 타이어 공기압, 코너링 속도 데이터를 실시간으로 계측해 주는 수많은 센서처럼, 헤비 리프터들이 쌓아 올린 스트렝스를 단 1%의 유실도 없이 바벨에 전달하도록 돕는 프라이빗 **바이오메카닉스 텔레메트리 센서(Biomechanics Telemetry Analyzer)**를 유저의 기기 자원만을 활용하는 클라이언트 단 AI로 구현합니다.

---

## 1. 아키텍처 개요 (Architecture Overview)

```
[유저 웹캠] ──▶ [Browser (React + MediaPipe)] ──▶ (실시간 역학 연산 및 UI 텔레메트리 렌더링)
                         │
                         ▼ (계측 JSON 데이터 전송)
                 [Supabase (BaaS)] ──▶ (인증 및 세션 텔레메트리 로그 저장)
```

---

## 2. 타겟 시장 및 페인 포인트 (Target Market & Pain Point)

### 2.1 핵심 페인 포인트
1. **역학적 힘의 누수 (Energy Leakage)**: 스쿼트, 데드리프트(SBD) 등 고중량 운동 시 미세한 골반 축 무너짐(스쿼트 벗윙크)이나 요추 굴곡(데드리프트 척추 굽음)으로 인해 힘의 작용선이 어긋나 스트렝스 전달에 비효율(누수)이 발생함.
2. **참견식 코칭에 대한 거부감**: 이미 높은 숙련도를 갖춘 헤비 리프터들은 자세를 훈계하는 선생이 아닌, 정밀 데이터(각도 편차, 모멘트 암의 상태)를 건네주는 계측 도구를 강력히 요구함.
3. **텔레메트리 데이터의 지연**: 녹화본 프레임 확인은 세트 수행 중 즉각적인 미세 교정 피드백을 전달받을 수 없게 만듦.

### 2.2 타겟 유저
* **Primary 타겟**: 파워리프팅, 스트렝스 트레이닝을 수행하며 바벨로의 동력 손실을 최소화하려는 SBD 헤비 리프터 및 코어 리프터층.
* **진입 장벽**: 30kg 수준의 역도성 운동에 대한 깊은 해부역학적 이해와 React 기반의 클라이언트 추론 기술이 융합된 뾰족한 도메인 결합성.

## 3. 기술 스택 (Technology Stack)

* **Client-side AI & Vision**: `TensorFlow.js` + `MediaPipe Pose` (또는 `MoveNet`)
  * *이유*: 서버로 영상을 전송하지 않고 유저의 브라우저 단에서 직접 33개의 관절 좌표(Landmarks)를 추출. GPU 서버 비용을 완전히 절감할 수 있는 핵심 기술.
* **Frontend Framework**: `React.js`
  * *이유*: 실시간 비디오 스트림 위에 스켈레톤(Skeleton) 레이어를 얹고, 각도 변화에 따라 UI를 동적으로 변경(예: 위험 시 빨간색 표시)하기에 가장 최적화된 컴포넌트 구조.
* **Styling**: Vanilla CSS (또는 Tailwind CSS)
  * *이유*: 생산성을 극대화하여 1인 개발 시 UI 디자인 및 퍼블리싱 시간을 최소화.
* **Backend & DB**: `Supabase`
  * *이유*: PostgreSQL 기반의 BaaS로, 백엔드 API 구축 리소스를 최소화하며 계측된 관절 궤적 로그(JSON)를 유저 계정별로 영구 동기화.
* **Development Tools**: `Cursor Editor` & `Antigravity` (AI-powered)
  * *이유*: MediaPipe 연동 등 복잡한 프론트엔드 보일러플레이트 코드 작성 및 디버깅 시 AI의 보조를 받아 개발 속도를 극대화.

## 4. MVP 핵심 기능 정의 (Minimum Viable Product Features)

초기 MVP 단계에서는 모바일 앱스토어 심사나 화려한 UI 대신, **스쿼트의 '벗윙크(Butt-wink) 골반 축 이탈' 및 데드리프트의 '요추 굴곡' 계측**에 집중하여 문제를 해결하는 단일 웹 링크를 구축합니다.


| 기능 분류 | 상세 기능 | 비고 |
| :--- | :--- | :--- |
| **실시간 캡처** | 웹캠/스마트폰 카메라 스트림 연동 | 브라우저 권한 획득 필요 |
| **관절 추적** | MediaPipe 기반 실시간 요추-골반-무릎 좌표 추출 | 클라이언트 연산 |
| **자세 분석** | 스쿼트 깊이(수평 이하) 및 벗윙크(골반 말림) 감지 로직 | 수학적 각도 산출 규칙 적용 |
| **실시간 피드백** | 화면 내 스켈레톤 색상 변경 (정상: 녹색 / 위험: 적색) | 시각적 즉각 알림 |
| **리포트 (저장)** | 운동 종료 후 세트별 성공 횟수 및 주의점 요약 데이터 저장 | Supabase DB 연동 (또는 로컬 스토리지 대체) |

## 5. 5주 개발 로드맵 (5-Week Roadmap)

* **1주 차: 기획 구체화 및 알고리즘 설계**
  * 스쿼트 수행 시 부상과 직결되는 핵심 지표 정의 (예: 요추 각도와 대퇴골 각도의 상호 연관성).
  * 화면 설계 및 핵심 인터페이스 설계.
* **2~3주 차: AI 파이프라인 및 프론트엔드 구축**
  * React 프로젝트 세팅 및 웹캠 컴포넌트 구현.
  * MediaPipe Pose/MoveNet 라이브러리를 이식하여 실시간 관절 좌표 추출 테스트 및 프레임 드랍(FPS) 최적화.
* **4주 차: 피드백 알고리즘 구현 및 백엔드 연동**
  * 추출된 좌표 데이터를 기반으로 엉덩이-무릎-발목 각도를 계산하는 수학적 로직 구현.
  * Supabase를 연결하여 유저 인증 및 간단한 운동 결과 테이블 스키마 구축.
* **5주 차: 배포 및 필드 테스트**
  * Vercel 또는 Netlify를 통해 무료 웹 호스팅 배포.
  * 피트니스 커뮤니티 및 지인 리프터 10~20명을 대상으로 MVP 링크 배포 및 피드백 수집.

## 6. 비디오 품질 예외 처리 방안 (Garbage In, Garbage Out 예외 처리)

아무리 분석 알고리즘이 정교해도 원본 비디오 품질이 낮으면 잘못된 피드백이 제공됩니다. 복잡한 AI 화질 개선 모델 대신 UX 및 클라이언트 단 예외 처리로 방어합니다.

### 6.1 기술적 방어: MediaPipe '신뢰도(Confidence Score)' 활용
* MediaPipe가 제공하는 keypoints의 `score` 또는 `visibility` (0.0 ~ 1.0)를 사용합니다.
* **임계값(Threshold) 설정**: 골반(Hips), 무릎(Knees), 발목(Ankles) 등 주요 관절의 신뢰도가 0.5 미만으로 떨어질 경우, 화면에 경고 문구("관절이 명확히 보이지 않습니다. 카메라 앵글을 확인해주세요")를 노출하고 측정을 일시 정지시킵니다.

### 6.2 UX의 제약 조건 명확화 (좁고 깊게)
* **단일 화각 강제**: MVP 단계에서는 **"측면 90도에서 촬영한 영상만 분석 가능합니다"**로 범위를 제한하여 기하학적 연산 복잡도를 낮춥니다.
* **복장 가이드**: "무릎과 골반 라인이 잘 보이는 타이트한 복장을 권장합니다"라는 안내 및 체크리스트를 촬영 시작 전 필수 팝업으로 배치합니다.
